# Архитектура

*[English version](ARCHITECTURE.md)*

## Поток данных
```
response (dict/json) --> validator.validate(contract) --> ValidationResult
                                                                |
                                                                v
                                                   storage.record(result, ...)
                                                                |
                            +-----------------------------------+-------------------------------+
                            |                                                                     |
                            v                                                                     v
                storage.baseline_and_recent()                                     test_generator.generate_tests()
                            |                                                                     |
                            v                                                                     v
             drift.DriftDetector.compare() -> DriftReport                          pytest regression file (text)
```

`ci_gate.run_gate()` оркеструет всё это для пакета ответов: проверяет каждый, записывает в хранилище, проверяет бюджеты по токенам/латентности/стоимости и (если в хранилище достаточно истории) просит детектор дрейфа сравнить недавнее окно прогонов со старым базовым окном. Итоговый `GateResult` предоставляет единый булев флаг `.ok` и `.exit_code()`, поэтому он встраивается прямо в любую CI-систему как shell-шаг.

## Модули

| Модуль | Ответственность |
|---|---|
| `contract.py` | Разбор YAML/JSON-файла контракта в dataclass `Contract` |
| `validator.py` | Проверка одного ответа против `Contract`, правило за правилом |
| `budget.py` | Проверка токенов/латентности/стоимости против бюджета контракта |
| `storage.py` | Сохранение каждого прогона в SQLite; вычисление сводок по окнам |
| `drift.py` | Сравнение двух окон `RunSummary` и обнаружение регрессий |
| `test_generator.py` | Превращение сохранённых сбоев в pytest-файл |
| `ci_gate.py` | Объединение валидатора + бюджета + хранилища + дрейфа для пакета |
| `reporters.py` | Рендер `GateResult` в text / JSON / JUnit XML |
| `config.py` | Централизованная конфигурация через переменные окружения |
| `cli.py` | Команды командной строки `validate` / `ci-gate` / `gen-tests` |

## Принципы проектирования

1. **Каждый провал правила объясним.** Никакого непрозрачного «показателя качества» — каждое правило сообщает свой pass/fail и сообщение.
2. **История хранится в одном месте (SQLite)**, поэтому обнаружение дрейфа и генерация тестов читают из одного источника истины.
3. **Никаких сетевых вызовов в базовой библиотеке.** `contract_guard` наблюдает и оценивает уже имеющиеся у вас ответы; получение этих ответов от вашего LLM-провайдера — задача вашей интеграции, а не этой библиотеки.
4. **CI прежде всего.** Основной интерфейс — CLI-команда с осмысленным exit-кодом, поэтому она работает с любой CI-системой без дополнительной обвязки.
